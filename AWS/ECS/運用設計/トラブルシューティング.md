# ① ログ設計

## サービス概要

### CloudWatch Logs
- 小規模・シンプル構成向け
- 保持期間設定が可能
- サブスクリプションフィルターで、指定した文字列が含まれているログのみを抽出可能
  - 抽出したログをLambda経由でSNSやQ Developer in chat applicationsと連携しSlackなどに通知可能

### FireLens(Fluent Bit)
- サイドカー構成でログ転送
- Data Firehoseと連携してAWSサービス(S3、Redshift、Opensearch Service)や、サードパーティ(Splunk、Snowflake)へログ転送可能
- ログの振り分け制御が可能

## ログ管理手法の比較
| 手法 | 特徴 | デメリット |
|------|------|----------|
| CloudWatch Logs → S3エクスポート | シンプル | 最大12時間遅延 / コスト大 |
| サブスクリプションフィルター + Data Firehose | リアルタイム転送 | CloudWatch Logs + Data Firehoseでコスト最大 |
| FireLens（Fluent Bit） | 出力先分岐・同時転送 | 設計が複雑 / コスト小 |

## 設計判断
### 小規模・シンプル運用
- CloudWatch Logs
### 大規模 / 大量ログ・コスト最適化
- FireLens（Fluent Bit）
  - ログの振り分けが可能
  - 不要なログのCloudWatch出力を抑止

# ② メトリクス設計

| 手法 | 取得粒度 | 特徴 | コスト |
|------|----------|------|--------|
| 標準CloudWatchメトリクス | クラスタ / サービス | CPU・メモリの基本監視 | 低（追加コストなし） |
| Container Insights | クラスタ / サービス / タスク | タスク単位での監視（平均値）<br>ディスク・ネットワークも取得 | 中 |
| Container Insights observability | クラスタ / サービス / タスク / コンテナ | コンテナ単位で詳細分析が可能 | 高 |

## 設計判断
### 開発環境（最低限の監視）
- **標準CloudWatchメトリクスを選定**
### 検証環境 / 本番環境（異常検知・運用）
- **Container Insightsを選定**
  - タスク単位で状態把握が可能
### 詳細分析・トラブルシュートが必要な場合
- **Container Insights Enhanced observability**
  - コンテナ単位でボトルネック分析が可能

# ③ 設計上の注意事項
## スケール時の影響
- サイドカーコンテナが複数の場合、スケールアウトする際、アプリケーションコンテナだけでなく、サイドカーコンテナも共に増加
- ECSタスクがスケールアウトするたび、消費するリソースも増え、Container Insights enhanced observabilityを有効化している場合、サイドカーコンテナ分のカスタムメトリクス取得が大幅に増加

## 対策
- スケールアウトが多数発生する場合、サイドカーコンテナをECSサービスとして切り離すことで、必要なリソース量を制御しながらスケールアウトが可能
